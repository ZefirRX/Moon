# Учебная практика: разработка индивидуального приложения и ведением проекта в GitLab

## 1. Общая характеристика практики

**Вид практики:** учебная практика.  
**Объём:** 108 часов.  
**Формат выполнения:** индивидуально.  
**Категория обучающихся:** 2 курс, программисты.  
**Предполагаемый уровень подготовки:** средний.  
**Допустимые языки и технологии:** Python, C# или иной стек по согласованию с преподавателем.  
**Допустимые GUI-технологии:** tkinter, PyQt/PySide, WxPython, Pygame, WPF, Windows Forms и другие GUI-фреймворки по согласованию.  
**Допустимые форматы внешних данных:** JSON, CSV, TXT для простых логов, SQLite по согласованию.  
**Итог практики:** готовое десктопное приложение, исходный код, внешние файлы данных, ресурсы, сборка, проектная документация в GitLab Wiki, задачи в GitLab Issues, история коммитов и финальная сдача файлов.

Учебная практика направлена на разработку индивидуального десктопного приложения с графическим интерфейсом, внешним хранением данных и расширенным набором пользовательских функций. Тематика приложения выбирается студентом самостоятельно или определяется по согласованию с преподавателем. Приложение может быть обучающим, справочным, расчётным, учётным, тренировочным, информационным, аналитическим, мультимедийным или иным прикладным программным продуктом.

Ключевое требование: приложение не должно быть программой из одного окна и одной функции. Оно должно иметь понятную пользовательскую логику, несколько экранов или окон, работу с внешними данными, сохранение результатов действий пользователя, обработку ошибок, структуру проекта и собранную запускаемую версию.

Практика организуется как полный мини-цикл разработки программного продукта: анализ задачи, проектирование, разработка интерфейса, проектирование внешних данных, реализация, тестирование, сборка и подготовка проекта к сдаче. Работа над проектом ведётся в GitLab.

---

## 2. Цель и задачи практики

### 2.1. Цель практики

Цель практики — сформировать у студента практический опыт разработки законченного индивидуального программного продукта с графическим интерфейсом, внешним хранением данных и управлением проектом через GitLab.

### 2.2. Задачи практики

В ходе практики студент должен:

1. Выбрать и обосновать тему индивидуального приложения.
2. Описать назначение приложения и пользовательские задачи.
3. Сформулировать техническое задание.
4. Спроектировать пользовательские сценарии и карту экранов.
5. Подготовить макеты основных экранов.
6. Спроектировать структуру внешних данных.
7. Спроектировать архитектуру приложения.
8. Реализовать приложение.
9. Реализовать обработку ошибок и некорректного ввода.
10. Провести тестирование приложения.
11. Выполнить сборку приложения.
12. Вести проект в GitLab: репозиторий, Wiki, Issues, Milestones, Board, коммиты.
13. Подготовить итоговый комплект файлов для сдачи.

## 3. Общие требования к итоговому приложению

Итоговое приложение должно быть десктопным программным продуктом с графическим интерфейсом. Оно должно решать выбранную прикладную задачу и обеспечивать взаимодействие пользователя с данными через интерфейс.

### 3.1. Обязательные признаки приложения

Приложение должно иметь:

1. Графический интерфейс.
2. Несколько экранов, окон или представлений.
3. Главное меню или иную систему навигации.
4. Внешнее хранение данных.
5. Загрузку данных из внешнего файла.
6. Отображение данных в интерфейсе.
7. Добавление, изменение, выбор, расчёт, проверку или иную активную работу с данными.
8. Сохранение результата работы пользователя.
9. Поиск, фильтрацию, сортировку, выборку, историю, статистику или другой дополнительный механизм работы с данными.
10. Обработку некорректного ввода.
11. Обработку ошибок при работе с файлами.
12. Информационные сообщения для пользователя.
13. Собранную запускаемую версию приложения.

### 3.2. Возможные направления проектов

Допускаются разные типы приложений:

- обучающее приложение;
- справочник;
- каталог;
- система учёта;
- интерактивный помощник;
- тренажёр;
- приложение для тестирования;
- приложение для работы с наборами данных;
- приложение для подбора и фильтрации информации;
- тематический симулятор;
- мультимедийное информационное приложение;
- приложение для анализа или визуализации данных.

Тематика может быть учебной, профессиональной, прикладной или свободной, если она позволяет реализовать полноценное GUI-приложение с набором функций и внешним хранением данных.

## 4. Обобщённое техническое задание

### 4.1. Наименование работы

**Разработка индивидуального десктопного приложения.**

### 4.2. Назначение приложения

Разрабатываемое приложение предназначено для решения выбранной прикладной задачи. Тематика определяется студентом самостоятельно или выбирается по согласованию с преподавателем. Приложение должно обеспечивать удобное взаимодействие пользователя с данными через графический интерфейс.

### 4.3. Общие требования

Приложение должно:

- быть десктопным;
- иметь графический интерфейс;
- содержать несколько экранов или окон;
- использовать внешние файлы данных;
- обеспечивать загрузку данных;
- обеспечивать сохранение данных;
- включать несколько пользовательских функций;
- иметь понятную навигацию;
- обрабатывать ошибки;
- иметь собранную запускаемую версию;
- сопровождаться проектной документацией;
- разрабатываться с использованием GitLab.

### 4.4. Требования к интерфейсу

Интерфейс должен быть понятным, логичным и пригодным для пользователя.

В приложении *рекомендуется* предусмотреть:

1. Стартовый экран или главное окно.
2. Главное меню или панель навигации.
3. Основной рабочий экран.
4. Экран просмотра данных.
5. Экран добавления или редактирования данных.
6. Экран поиска, фильтрации или выбора.
7. Экран результата, статистики, истории или состояния.
8. Экран справки, настроек или информации о приложении.

Элементы управления должны иметь понятные названия. Пользователь должен понимать, что произойдёт после нажатия кнопки. Действия, которые могут привести к потере данных, должны сопровождаться подтверждением или предупреждением.

### 4.5. Требования к внешнему хранению данных

Приложение должно использовать внешние файлы данных. Данные не должны полностью храниться внутри исходного кода.

Минимальные требования:

- данные хранятся отдельно от исходного кода;
- при запуске приложение загружает данные из файла;
- пользователь может работать с данными через интерфейс;
- изменения могут быть сохранены во внешний файл;
- приложение проверяет наличие файлов данных;
- приложение сообщает пользователю об ошибках чтения и записи.

Допустимые форматы:

- JSON как основной формат структурированных данных;
- CSV для табличных данных или экспорта;
- TXT для простых логов;
- SQLite по согласованию с преподавателем.

### 4.6. Требования к функциональности

Минимальный набор функций приложения:

1. Запуск приложения.
2. Переход между экранами.
3. Загрузка данных из внешнего файла.
4. Отображение данных в интерфейсе.
5. Добавление или изменение данных.
6. Сохранение данных.
7. Поиск, фильтрация или сортировка.
8. Просмотр подробной информации.
9. Проверка корректности ввода.
10. Вывод сообщений пользователю.
11. Сохранение результата, истории или состояния.
12. Корректное завершение работы.

### 4.7. Дополнительные функции

Для повышенного уровня студент может реализовать дополнительные возможности:

- расширенную статистику;
- диаграммы;
- экспорт данных;
- импорт данных;
- настройки интерфейса;
- тёмную/светлую тему;
- пользовательские профили;
- генерацию данных;
- историю действий;
- журнал событий;
- мультимедийные элементы;
- резервное копирование данных;
- расширенную проверку данных;
- интерактивную справку;
- таймер;
- систему достижений;
- расширенный поиск;
- работу с несколькими наборами данных;
- случайную генерацию заданий или записей;
- сортировку по нескольким параметрам.

### 4.8. Требования к обработке ошибок

Приложение должно корректно обрабатывать:

- отсутствие файла данных;
- пустой файл;
- повреждённый файл;
- некорректный формат данных;
- некорректный ввод пользователя;
- попытку сохранить неполные данные;
- ошибку доступа к файлу;
- попытку выполнить действие без выбранного объекта;
- невозможность загрузить изображение или другой ресурс;
- ошибку при сохранении результата.

Сообщения об ошибках должны быть понятны пользователю и, по возможности, содержать рекомендацию по исправлению проблемы.

### 4.9. Требования к сборке

По итогам практики студент должен предоставить собранную версию приложения.

Для Python допускается сборка через PyInstaller или аналогичный инструмент. Для C# допускается сборка через Visual Studio в режиме Release или публикация приложения.

Собранное приложение должно запускаться без открытия среды разработки. Если для запуска нужны дополнительные файлы данных или ресурсы, они должны быть размещены рядом с приложением в понятной структуре папок.

## 5. Проектные документы

Проектные документы ведутся преимущественно в GitLab Wiki. Они должны обновляться по мере разработки, а не заполняться только в конце практики.

### 5.1. Паспорт проекта

Паспорт проекта — краткий документ, фиксирующий общий замысел индивидуального приложения.

В паспорте проекта должны быть:

1. Название проекта.
2. ФИО студента и группа.
3. Краткое описание идеи приложения.
4. Цель разработки.
5. Целевая аудитория.
6. Основные пользовательские задачи.
7. Выбранный язык программирования и инструмент разработки.
8. Планируемый тип интерфейса.
9. Ожидаемый итоговый результат.

В паспорте проекта не требуется указывать:

- формат хранения данных;
- подробную структуру файлов;
- подробную структуру функций;
- конкретные разделы обучающего курса;
- наличие тестов;
- наличие тренажёра.

Эти сведения должны быть отражены в техническом задании, модели данных, функциональной декомпозиции и архитектуре.

Шаблон паспорта проекта:

```markdown
# Паспорт проекта

## 1. Название проекта

## 2. Автор проекта

## 3. Группа

## 4. Краткое описание идеи приложения

## 5. Цель разработки

## 6. Целевая аудитория

## 7. Основные пользовательские задачи

## 8. Язык программирования и среда разработки

## 9. Планируемый тип интерфейса

## 10. Ожидаемый итоговый результат
```

### 5.2. Техническое задание

Техническое задание — основной документ проекта. Оно описывает, что должно делать приложение и по каким признакам его можно проверить.

В ТЗ должны быть:

1. Назначение приложения.
2. Описание предметной области.
3. Пользовательские сценарии.
4. Функциональные требования.
5. Требования к интерфейсу.
6. Требования к внешнему хранению данных.
7. Требования к обработке ошибок.
8. Требования к сборке.
9. Требования к качеству реализации.
10. Перечень функций базового уровня.
11. Перечень функций повышенного уровня.

### 5.3. Пользовательские сценарии

Студент должен описать основные действия пользователя в приложении.

Примеры сценариев:

```text
Сценарий 1. Запуск приложения и переход к основному экрану.
Сценарий 2. Просмотр списка объектов.
Сценарий 3. Добавление новой записи.
Сценарий 4. Редактирование существующей записи.
Сценарий 5. Поиск или фильтрация данных.
Сценарий 6. Сохранение результата.
Сценарий 7. Просмотр истории или статистики.
Сценарий 8. Завершение работы приложения.
```

### 5.4. Карта экранов / User Flow

Студент должен подготовить схему переходов между экранами приложения.

Обобщённый пример:

```text
Стартовый экран
        |
        v
Главное меню
   |          |           |             |
   v          v           v             v
Просмотр   Добавление   Поиск/       Настройки
данных     данных       фильтрация
   |          |           |
   v          v           v
Карточка   Сохранение   Результаты
объекта    записи       поиска
   |
   v
Редактирование
```

Карта экранов должна соответствовать реальной навигации в готовом приложении.

### 5.5. Макеты интерфейса

Студент должен подготовить макеты основных экранов. Макеты могут быть выполнены в draw.io, Figma, PowerPoint, графическом редакторе.

Минимально рекомендуется подготовить макеты:

1. Стартового экрана.
2. Главного меню.
3. Основного рабочего экрана.
4. Экрана добавления или редактирования данных.
5. Экрана просмотра подробной информации.
6. Экрана поиска или фильтрации.
7. Экрана результата, статистики или истории.
8. Экрана ошибки, справки или настроек.

### 5.6. Модель данных

Модель данных — обязательный документ, потому что приложение должно работать с внешними данными.

В документе нужно указать:

- какие файлы данных используются;
- для чего нужен каждый файл;
- какие поля есть в каждом файле;
- какие поля обязательны;
- какие типы данных используются;
- как данные загружаются;
- как данные сохраняются;
- какие ошибки возможны при работе с файлами;
- какие примеры данных используются для тестирования.

Пример структуры папки данных:

```text
data/
├── items.json       — основные записи приложения
├── users.json       — данные пользователей, если используются
├── history.json     — история действий или результатов
├── settings.json    — настройки приложения
└── export.csv       — экспортированные данные
```

Пример универсального JSON:

```json
{
  "items": [
    {
      "id": 1,
      "title": "Название объекта",
      "description": "Описание объекта",
      "category": "Категория",
      "tags": ["пример", "данные"],
      "created_at": "2026-05-11",
      "updated_at": "2026-05-11"
    }
  ]
}
```

### 5.7. Архитектура приложения

Студент должен описать внутреннюю структуру проекта: модули, классы, сервисы и связи между ними.

Главное требование: нельзя выполнять весь проект в одном файле без структуры. Интерфейс, данные и логика должны быть хотя бы минимально разделены.

### 5.8. Функциональная декомпозиция

Функциональная декомпозиция показывает, из каких функциональных модулей состоит приложение.

Пример:

```text
Приложение
├── Модуль интерфейса
├── Модуль загрузки данных
├── Модуль сохранения данных
├── Модуль просмотра объектов
├── Модуль добавления и редактирования
├── Модуль поиска и фильтрации
├── Модуль статистики или истории
├── Модуль настроек
└── Модуль обработки ошибок
```

### 5.10. Инструкция по сборке и запуску

Обязательный документ.

Для Python:

```text
1. Установить Python.
2. Установить зависимости из requirements.txt.
3. Запустить main.py.
4. Для сборки использовать PyInstaller.
5. Проверить запуск exe-файла из папки dist.
```

Для C#:

```text
1. Открыть проект в Visual Studio.
2. Проверить наличие папок Data и Resources.
3. Выполнить сборку Release.
4. Опубликовать проект.
5. Проверить запуск exe-файла без среды разработки.
```

## 6. GitLab как обязательная среда управления проектом

Работа над проектом должна вестись в GitLab. GitLab используется не только как хранилище исходного кода, но и как среда управления учебным проектом.

### 6.1. Обязательные элементы GitLab-проекта

В GitLab-проекте студента должны быть:

1. Репозиторий с исходным кодом.
2. README.md.
3. Внешние файлы данных.
4. Ресурсы приложения.
5. GitLab Wiki с проектной документацией.
6. GitLab Issues с задачами по этапам практики.
7. GitLab Milestones для контрольных блоков.
8. Issue Board для отслеживания статуса задач.
9. Регулярные коммиты.
10. Финальная сборка приложения.
11. Инструкция по запуску.
12. Финальный tag или release, если используется.

### 6.2. Роль репозитория

В репозитории должны храниться:

- исходный код;
- внешние файлы данных;
- ресурсы приложения;
- README;
- файл зависимостей;
- файл проекта;
- инструкция по запуску;
- финальная сборка или ссылка на неё;
- при необходимости копии проектных документов.

### 6.3. Роль GitLab Wiki

GitLab Wiki используется как основное место проектной документации.

Рекомендуемая структура Wiki:

```text
Home
├── 01. Паспорт проекта
├── 02. Техническое задание
```

Wiki должна обновляться по мере разработки. Недопустимо полностью заполнять Wiki только в конце практики без связи с этапами работы.

### 6.4. Роль Issues

GitLab Issues используются для постановки и отслеживания задач.

Каждая задача должна иметь:

- название;
- описание;
- ожидаемый результат;
- Definition of Done;
- label;
- milestone;
- статус на доске.

Минимальные типы задач:

```text
[DOC] Документация
[UI] Интерфейс
[DATA] Данные
[CORE] Основная логика
[TEST] Тестирование
[BUILD] Сборка
[BUG] Ошибка
[REFACTOR] Улучшение кода
[FEATURE] Новая функция
[SETUP] Настройка проекта
[ARCH] Архитектура
```

Шаблон issue:

```markdown
## Описание задачи

Что необходимо сделать.

## Ожидаемый результат

Что должно появиться после выполнения задачи.

## Definition of Done

- [ ] Результат реализован
- [ ] Изменения добавлены в репозиторий
- [ ] При необходимости обновлена Wiki
- [ ] Проверена работа приложения
- [ ] Задача готова к проверке
```

Пример issue:

```markdown
# [DATA] Спроектировать структуру внешних JSON-файлов

## Описание задачи

Необходимо определить, какие данные приложение хранит во внешних файлах, какие поля используются, какие поля обязательны, как выполняется загрузка и сохранение.

## Ожидаемый результат

- создана страница Wiki "Модель данных";
- добавлены примеры JSON-файлов;
- создана папка data/;
- приложение в дальнейшем сможет использовать эту структуру.

## Definition of Done

- [ ] описаны все файлы данных;
- [ ] указаны обязательные поля;
- [ ] приведён пример заполнения;
- [ ] файл добавлен в репозиторий;
- [ ] задача закрыта после проверки.
```

### 6.6. Роль Issue Board

Issue Board используется как доска задач.

Рекомендуемые колонки:

```text
Backlog
Ready
In Progress
Review
Done
```

Дополнительные колонки:

```text
Blocked
Bug
```

Логика работы:

- **Backlog** — задача создана, но ещё не готова к выполнению.
- **Ready** — задача понятна и готова к работе.
- **In Progress** — студент выполняет задачу.
- **Review** — задача выполнена и ждёт проверки.
- **Done** — задача принята.
- **Blocked** — задача заблокирована проблемой.
- **Bug** — ошибка, требующая исправления.

### 6.7. Требования к коммитам

Коммиты должны отражать ход разработки.

Требования:

1. Коммиты должны быть регулярными.
2. Один коммит должен отражать одно логическое изменение.
3. Недопустим один финальный коммит со всем проектом.
4. Сообщение коммита должно быть понятным.
5. Желательно связывать коммиты с issues.
6. В коммитах не должно быть временного мусора, лишних архивов, файлов кэша и системных файлов.

Примеры сообщений коммитов:

```text
docs: add project passport
docs: add technical task
ui: create main menu screen
ui: add navigation between screens
data: add json loader
data: implement save service
feature: add item search
feature: add edit form
test: add manual test plan
build: add pyinstaller config
fix: handle missing data file
```

## 7. Траектория практики

### Этап 1. Вводный этап и выбор идеи проекта

**Цель:** определить общий замысел индивидуального приложения.

Студент выбирает тему проекта, формулирует проблему или пользовательскую задачу, определяет целевую аудиторию и выбирает технологический стек.

**Результат этапа:**

- выбранная тема;
- сформулированная идея приложения;
- черновое описание назначения;
- выбранный язык и GUI-фреймворк;
- создан GitLab-проект;
- создан README.md;
- создана начальная структура репозитория.

**GitLab-результат:**

- создан репозиторий;
- создана Wiki-страница «Паспорт проекта»;
- создан milestone `M1. Инициация проекта`;
- созданы первые issues.

**Примерные issues:**

```text
[DOC] Заполнить паспорт проекта

[SETUP] Создать структуру репозитория
```

**Контрольная точка:** паспорт проекта и созданный GitLab-проект.

**Результат этапа:**

- описание предметной области;
- список пользовательских задач;

- предварительный список функций.

**GitLab-результат:**

- создана Wiki-страница «Пользовательские сценарии»;
- созданы задачи на функции приложения;
- сформирован первичный backlog.

**Примерные issues:**

```text

[FEATURE] Сформировать список базовых функций
```
